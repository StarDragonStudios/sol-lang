# Sol C backend

## Purpose

The C backend transforms a semantically analyzed Sol executable into
deterministic C source code.

The backend is an implementation phase of the compiler. It does not define
Sol language semantics and does not repeat parsing, name resolution, type
checking, module injection resolution, or executable entry-point discovery.

## Input contract

C generation accepts a `SemanticProgramAnalysisResult` produced by executable
semantic analysis.

The semantic program must:

- contain one resolved `ProgramEntryPoint`
- contain no program-level error diagnostics
- contain no module-level error diagnostics
- retain canonical module and function symbols
- contain completed, frozen semantic scopes

The backend rejects invalid input with `CCodeGenerationException`. This
exception represents a compiler-pipeline contract violation rather than a new
Sol source diagnostic.

## In-memory output

Generation returns an immutable `CTranslationUnit`.

A translation unit:

- stores the complete generated C source in memory
- uses `\n` as its line ending
- ends non-empty source with exactly one newline
- performs no filesystem writes
- does not invoke a native C compiler

Writing `.c` files belongs to the CLI and toolchain layer.

## Deterministic formatting

Generated C source follows these rules:

- four spaces per indentation level
- no tab indentation
- no trailing whitespace
- no accidental leading blank lines
- no repeated trailing blank lines
- exactly one final newline for non-empty output
- no timestamps
- no absolute source paths
- no locale-dependent or environment-dependent content

Each generation call owns fresh writer state. Failed and concurrent generation
calls do not share mutable state.

## Initial translation-unit scaffold

Issue #50 establishes the backend without lowering Sol declarations yet.

The initial generated source is:

```c
/*
 * Generated by the Sol compiler.
 * Do not edit manually.
 */

typedef int sol_generated_translation_unit;
```

The placeholder typedef keeps the generated translation unit non-empty. Later
backend work may remove it once actual declarations are emitted.

## Pipeline boundaries

The C backend is separate from:

- source-file discovery
- lexical analysis
- parsing
- semantic analysis
- filesystem output
- native compiler discovery
- native compiler invocation
- object generation
- linking
- executable execution
- the Sol runtime

## Backend roadmap

### Issue #50

Establishes:

- the public generation API
- semantic input validation
- immutable in-memory C output
- deterministic source writing
- the minimal generated scaffold

### Issue #51

Adds function, primitive expression, unary expression, binary expression, and
return-statement lowering.

### Issue #52

Adds local variable, constant, mutable local, and assignment lowering.

### Issue #53

Adds conditional, loop, and nested control-flow lowering.

### Issue #54

Adds function-call lowering and deterministic module and symbol name mangling.

Runtime headers, primitive runtime support, strings, CLI output, native
compilation, linking, and execution are handled by later roadmap phases.
